# 配置说明（Config Notes）

每个配置键写成一条完整的档案。模板字段如下，缺失的字段标注「待确认」，不允许留空或猜测。

## 单键模板

```markdown
### `order.close.timeout-minutes`

- 配置键：`order.close.timeout-minutes`（保留原始 key，不翻译、不改大小写）
- 默认值：30（来源：`application.yml` 第 42 行）/ 未找到默认值 → 标记「待确认：需查配置中心」
- 单位：分钟（必须写明；歧义单位是支付配置事故的高发源）
- 作用域：商户级，渠道级配置可覆盖（来源：`ConfigResolver#resolve` 的查找顺序）
- 启用条件：仅当开关 `order.close.enabled=true` 且订单状态为 `PENDING` 时生效
- 风险：调小 → 用户支付中订单被提前关闭，引发掉单投诉；调大 → 库存占用时间变长
- 测试环境验证：创建订单后等待超时窗口，确认订单状态变为 `CLOSED` 且发出 `order.closed` 事件
```

## 字段要求

- **配置键**：保留原始 key。涉及多套命名（yml key、环境变量、配置中心 key）时全部列出并注明映射关系。
- **默认值**：必须写来源（文件+行号、`@Value` 默认、配置中心样例）。代码默认值与配置文件默认值不一致时，写出两者并说明实际生效顺序。
- **作用域**：全局 / 商户 / 渠道 / 环境 / 实验组。写明覆盖优先级链，这决定了排查时先查哪一层。
- **单位**：分钟、秒、毫秒、金额单位（分 vs 元）、百分比 vs 小数。金额与时间字段无单位即视为文档缺陷。
- **启用条件**：开关依赖、灰度分组、状态机入口、事件类型。一个配置键孤立看是对的、组合看才生效，组合条件必须写全。
- **风险**：误配后影响哪些行为——支付、订阅扣款、退款、关单、通知重发。按「调大会怎样 / 调小会怎样 / 关掉会怎样」三个方向各写一句。
- **测试环境建议**：写出能证明配置生效的最小操作序列；同时写明哪些现象**不能**作为充分证据（见下）。

## 不充分证据清单

以下现象常被误当作「配置已生效」的证据，单独出现时都不充分：

- 接口返回成功 → 只证明请求被接受，不证明走到了该配置控制的分支。
- 日志里出现配置值 → 可能是启动时打印的旧快照，不代表运行时热更新已生效。
- 测试环境行为正确 → 测试环境与生产的作用域覆盖链可能不同（商户级覆盖只配在生产）。

充分证据 = 触发该配置控制的具体分支，并观察到与配置值一致的行为差异（如改值前后超时窗口长度变化）。
