# 推荐结构与示例

支付/订阅文档的六段结构。每段都给出最小示例，照此填充即可得到可评审的初稿。

## 1. 术语表

保留英文枚举、事件名、配置 key 原文，中文只做解释，不做翻译替换——否则读者无法对照源码与日志。

| 原文 | 类型 | 中文解释 |
|---|---|---|
| `PENDING` | 订单状态枚举 | 已创建未支付，占用库存，受超时关单控制 |
| `invoice.payment_failed` | Webhook 事件 | 第三方扣款失败通知，触发重试或订阅降级 |
| `order.close.timeout-minutes` | 配置键 | 超时关单窗口，详见配置键一节 |

## 2. 场景矩阵

按「触发条件 → 入口 → 状态变化 → 外部事件 → 最终结果」组织，一行一个场景。矩阵的价值在于暴露没写到的格子——空格子就是待确认事项。

| 场景 | 触发条件 | 入口 | 状态变化 | 外部事件 | 最终结果 |
|---|---|---|---|---|---|
| 正常支付 | 用户完成支付 | 支付回调 `PayCallbackController` | `PENDING → PAID` | 发送 `order.paid` | 订单完成 |
| 超时关单 | 超过关单窗口未支付 | 定时任务 `OrderCloseJob` | `PENDING → CLOSED` | 发送 `order.closed` | 库存释放 |
| 关单与支付竞争 | 回调与关单同时到达 | 两个入口并发 | 以状态机守卫为准（源码：`OrderStateMachine`） | 待确认：是否触发退款 | 待确认 |

## 3. 配置键

按 config-notes 的单键模板逐键填写：默认值（带来源）、单位、作用域、启用条件、风险。

## 4. 验证点

四类分开列，并标注每条属于哪类证据（对应 source-backed-docs 的三级分类）：

- 源码检查：状态机守卫存在于 `OrderStateMachine#canClose`（源码已证实）。
- 配置检查：测试环境 `timeout-minutes=5`（配置已证实，来源：配置中心截图/导出）。
- 测试环境检查：创建订单等待 5 分钟，观察状态与事件（操作步骤，待执行）。
- 日志检查：关单时输出 `closing order {id}, window={n}m`，可用于回查线上行为。

## 5. 风险

逐条写清「误X → 后果」：误判（把竞争场景当正常关单）、误配（单位写错放大 60 倍）、重复通知（重试与首发都触发 webhook 处理）、状态回滚（人工改库绕过状态机守卫）、超时窗口边界（恰好在窗口边界支付的订单归属）。

## 6. 待确认事项

凡是需要业务方、运维、第三方后台或 DMS 人工查询才能确认的信息，集中列在文末，写明「向谁确认、确认什么、为什么文档内部无法自证」。例如：「线上是否已订阅 `invoice.payment_failed`——需第三方商户后台截图，源码只能证明存在处理路径（见 source-backed-docs）。」
