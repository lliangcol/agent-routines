# 源码背书的写作纪律（Source-Backed Docs）

文档中每一条断言必须归入三类之一，并随文标注。混淆类别——尤其是把推断写成事实——是支付文档最危险的缺陷，因为读者会基于它做线上操作决策。

## 三级证据分类

| 级别 | 含义 | 合格的证据形式 |
|---|---|---|
| 源码已证实 | 能从当前代码直接确认 | 代码路径+类/方法名（如 `WebhookHandler#onPaymentFailed`）、枚举定义、DTO 字段、状态机守卫、单元测试断言 |
| 配置已证实 | 能从配置文件、默认值、环境变量或配置中心样例确认 | 文件+行号、`@Value` 默认值、配置中心导出（注明环境与时间） |
| 运行环境待确认 | 需要真实环境、Webhook 订阅、第三方后台或人工执行结果 | 写成「待确认：向X确认Y」，不写结论 |

## 典型的越级错误

最常见的一类：**源码证明了处理能力，被写成了线上行为。**

- 源码存在 `invoice.payment_failed` 的处理器 → 只证明「收到该事件时会如何处理」。线上是否订阅了该事件，属于第三方后台配置，必须写「待确认」。
- 配置文件有默认值 30 → 只证明代码缺省。生产是否被配置中心覆盖为其他值，属于运行环境，写「待确认：查生产配置中心」。
- 测试里有重试三次的断言 → 证明代码逻辑。生产的重试开关是否开启，属于配置/运行环境。

判别方法：对每条断言问「如果只给我这份代码和配置、不给任何线上访问，我还能确认它吗？」不能 → 降级为待确认。

## 写法规范

- 每条断言标注来源：`（源码：OrderStateMachine#canClose）`、`（配置：application-prod.yml:18）`、`（待确认：需运维查询）`。
- 「待确认」不是弱化语气词，是一个段落类型：集中收录在文档末尾的待确认清单，写明向谁确认、确认什么。
- 源码证据引用当前 checkout 的真实路径，文档评审时可逐条点开核对；引用不存在的路径视为文档不合格。
- 已确认事项后续被验证时，把证据升级（待确认 → 配置已证实），并保留确认日期与来源。

## 失败案例

- 现象：文档写「支付失败后系统会自动降级订阅」，事故复盘发现线上从未订阅该 webhook。错误根源：把「存在处理器」写成了「线上会发生」。正确写法：「代码具备降级处理路径（源码：…）；线上订阅状态待确认（需第三方后台核实）」。
- 现象：文档默认值与生产实际值不符，按文档排障方向全错。错误根源：只查了代码缺省，未标注「生产可能覆盖」。正确写法：默认值标来源，并在作用域字段写明覆盖链。
