# Runtime Repair Checklist

Diagnose agent-runtime launch and hook failures layer by layer. The layers — PATH/shim, package tree, auth, version, encoding — fail independently; collapsing them into one guess produces the wrong repair.

## 1. Capture the Failure Exactly

Record together, before any hypothesis: which shell (PS 5.1 vs pwsh vs bash — `$PSVersionTable.PSVersion` / `echo $0`), the exact command line, full stderr verbatim, exit code, and the resolved executable path:

```powershell
(Get-Command claude -All).Source     # ALL resolutions, in order
```

```bash
command -v -a claude
```

"It doesn't start" is not a capture. The resolved path decides the whole diagnosis tree: a `.cmd`/`.ps1` shim under npm's prefix, a real binary, or a Store stub are three different problems.

## 2. Shim vs Real Binary

Package-manager-installed CLIs launch through a shim that locates the real package. Check the two layers separately:

```powershell
npm prefix -g                                    # where shims live
Get-Content "$(npm prefix -g)\claude.ps1"        # what the shim actually resolves
npm ls -g <package> --depth=0                    # is the package tree intact?
```

- Shim present + package tree broken (interrupted update, half-removed `node_modules`) → reinstall is the repair (gated: confirm first).
- Shim stale (points at a removed version directory) → the package manager's relink/reinstall, not PATH edits.
- Multiple shims from different managers (npm + a native installer) → PATH order decides which runs; report both paths before recommending removal of either.

## 3. Auth vs Launch

A binary that launches and then exits with an auth/login error is *not* a launch failure. Separate them by reading stderr: TLS/401/"please log in" → auth layer (browser login flows are user actions — suggest the user run the login command themselves); "module not found"/segfault/command not found → binary or package layer. Repairing the wrong layer (reinstalling over an auth problem) destroys evidence and fixes nothing.

## 4. Hook JSON Failures: Generate Bytes, Don't Quote Them

When a hook fails parsing its stdin payload, the suspect is shell quoting/encoding mangling the bytes (PS 5.1 BOM insertion, `echo` adding CRLF, quote stripping). Reproduce with a payload generated by a real language, bypassing shell quoting entirely:

```powershell
python -c "import json,subprocess,sys; p=subprocess.run(['the-hook'], input=json.dumps({'event':'test'}).encode(), capture_output=True); sys.stdout.write(str(p.returncode)+p.stderr.decode())"
```

If the generated-bytes payload works where the shell-quoted one fails, the defect is the invoking shell pipeline, not the hook. Check specifically: BOM (`Format-Hex` the first 3 bytes), CRLF vs LF, and PS 5.1's default non-UTF8 pipe encoding.

## 5. Version Triage Before Recommending Upgrades

```bash
claude --version                      # what runs locally
npm view <package> version            # latest on the registry
npm view <package> time --json        # release dates
```

Cross-check the registry and release notes before recommending an upgrade: confirm the observed bug is actually fixed in a newer release (changelog evidence), and check whether auto-update state explains a version mismatch (an auto-updated copy alongside a pinned one). "Upgrade and see" without changelog evidence is a guess wearing a repair's clothes.

## 6. Repair Boundaries

Readonly diagnosis needs no authorization. Gated, each occurrence: reinstalling packages, editing user config files, modifying PATH, disabling auto-update, editing hooks. Inspect user config (`~/.claude/settings.json` etc.) freely as evidence; propose edits as diffs for approval.

## Failure Cases

- Symptom: CLI works in one terminal, "not recognized" in another. Cause: PATH divergence between shells (machine vs user PATH, or a profile that prepends). Wrong response: reinstall. Correct response: diff `$env:Path` between the two terminals; the repair is one PATH entry, applied by the user.
- Symptom: hook works when run manually, fails when invoked by the runtime. Cause: different stdin encoding or environment in the runtime's invocation. Correct response: capture the runtime's actual payload bytes (tee to a file from inside the hook) and replay them with the generated-payload method.
- Symptom: after an auto-update, the CLI crashes on start. Correct response: check whether a pinned global install and the auto-updated copy now disagree (step 2's two-layer check); removing the wrong one loses the working copy.
